#  모의투자 웹 서비스 프로젝트 로드맵

## 1. 코딩 감각을 깨우는 단계별 백엔드 개발 및 작업 순서 (Milestones)

### v1.0: 기초 체력 기르기 (기본 CRUD 및 모의투자 기초 웹)
**목표:** 스프링 부트 구조에 익숙해지고, DB와 통신하는 기본기를 회복합니다.
- [x] **프로젝트 초기 설정:** Spring Boot 3.x, Java 21, Spring Data JPA, H2 (로컬 테스트용) / MySQL 설정.
- [ ] **회원 도메인 (User):** 회원 가입, 로그인 (Spring Security + JWT 기초), 회원가입 시 가상 예수금(예: 1,000만 원) 기본 지급 로직.
- [ ] **자산 도메인 (Asset):** 지원하는 가상 자산(비트코인, 이더리움 등) 메타 데이터 설계 및 등록.
- [ ] **기본 주문 도메인 (Order - REST API):** 시장가 매수/매도 API 구현. (이 단계에서는 실시간 시세가 아닌 DB에 하드코딩된 임의의 시세로 매매가 이루어짐).
- [ ] **지갑 도메인 (Wallet):** 유저별 보유 자산 및 잔여 예수금 조회.

### v2.0: 실시간 통신 연동 및 랭킹 시스템 (트래픽 대응 기초)
**목표:** 외부 API 연동과 대용량 데이터를 처리하는 최적화 방법을 학습합니다.
- [ ] **실시간 시세 연동:** 외부 가상자산 거래소(Upbit 등) API를 연동하여 실시간 시세를 백엔드에서 수집.
- [ ] **프론트엔드 실시간 통신:** 수집된 시세를 SSE(Server-Sent Events) 또는 WebSocket을 통해 프론트엔드로 실시간 푸시.
- [ ] **비동기 주문 체결 로직:** 지정가 주문 기능 추가 및 시세 도달 시 자동 체결되는 이벤트 기반 비동기 로직 구현.
- [ ] **수익률 계산 스케줄러:** 매일 자정 유저들의 총 자산(예수금 + 평가금액)을 스냅샷 찍어 주간/월간 수익률 계산.
- [ ] **랭킹 시스템 도입:** Redis Sorted Set을 활용하여 실시간 랭킹 산출 및 조회 API 구현.

### v3.0: AI 연동, 모니터링 및 상용 배포 (운영의 묘미)
**목표:** 실제 배포 후 발생할 수 있는 이슈를 방어하고, 서비스 가치를 높입니다.
- [ ] **AI 투자 성향 리포트:** 유저의 거래 내역(단타 위주, 장투 위주 등) 데이터를 프롬프트로 가공해 LLM(OpenAI API 등)에 전달하여 '당신의 투자 성향 리포트' 생성 로직 구현.
- [ ] **운영 인프라 구축:** AWS EC2, RDS에 Docker 기반 배포, GitHub Actions를 활용한 CI/CD 파이프라인 구성.
- [ ] **로깅 및 에러 트래킹:** Slf4j + Logback 기반 로깅, 에러 발생 시 Slack 알림 연동.
- [ ] **성능 테스트 및 튜닝:** JMeter를 활용한 더미 유저 1만명 접속 테스트, 병목 구간 파악 및 쿼리/인덱스 튜닝.

## 2. 유저 리텐션 및 랭킹 시스템 기획

**[리텐션(재방문) 유도 전략]**
- **초기화 티켓 시스템:** 수익률이 -50% 이하로 떨어지면 "파산 위기" 알림과 함께 한 달에 1번 계좌를 초기화할 수 있는 티켓을 주어 포기하지 않고 다시 시작하게 유도.
- **주간 리그전:** 전체 랭킹 외에 브론즈/실버/골드 등급별 '주간 리그'를 운영. 상위권 달성 시 프로필에 한정판 뱃지 부여.
- **AI 훈수 알림:** 뇌동매매(짧은 시간에 잦은 매매 후 손실) 발생 시 AI가 "진정하세요. 지금은 관망할 때입니다." 같은 재치 있는 푸시 알림/메시지 전송.

**[랭킹 시스템 구조]**
- **DB 부하 최소화:** 매번 DB를 `ORDER BY`로 조회하지 않음. Redis의 `ZSET(Sorted Set)`을 이용하여 유저의 ID와 수익률 점수를 메모리에 저장. `ZREVRANGE` 명령어로 O(log(N)) 시간 복잡도로 즉시 랭킹 제공.

## 3. 인프라 구축 및 운영 전략 (DevOps & 모니터링)

- **CI/CD 파이프라인:** 
  1. 개발자가 GitHub `main` 브랜치에 Push.
  2. GitHub Actions가 이를 감지하여 Java 빌드 및 테스트 수행.
  3. 성공 시 Docker Image를 빌드하여 Docker Hub에 Push.
  4. AWS EC2 서버에 접속하여 최신 Image를 Pull 받고 컨테이너 재시작.
- **로깅 (Logging):** 
  - `INFO`: 서비스의 주요 흐름 (주문 접수, 체결 완료).
  - `WARN`: 비정상적인 접근이나 잔액 부족 등의 비즈니스 예외.
  - `ERROR`: DB 연결 실패, 외부 API 타임아웃 등 개발자가 즉시 알아야 할 시스템 에러. (Slack Webhook 연동으로 알림 받기).

## 4. 면접관을 사로잡을 '의도된 트러블슈팅 시나리오' 3가지

1. **동시성 이슈 (Race Condition) 해결:**
   - *상황:* 한 유저가 동시에 "매수" 버튼을 빠르게 여러 번 클릭하거나, 다중 서버 환경에서 예수금 차감이 동시에 일어남. 예수금이 100만 원인데 100만 원어치 자산을 2번 사게 되는 버그 발생.
   - *해결:* DB 비관적 락(Pessimistic Lock) vs 낙관적 락(Optimistic Lock) vs Redis 분산 락(Redisson) 중 상황에 맞는 기술을 선택하고 그 이유를 증명.
2. **외부 API(시세) 연동 지연 및 장애 방어:**
   - *상황:* Upbit 등 외부 시세 API 서버가 잠시 다운되거나 응답이 10초 이상 지연되어, 우리 서버의 스레드까지 전부 대기 상태에 빠지고 서버가 다운됨.
   - *해결:* Spring Cloud Resilience4j를 이용한 서킷 브레이커(Circuit Breaker) 패턴 도입. 타임아웃 시 대체 로직(Fallback)으로 마지막 캐싱된 시세 보여주기.
3. **N+1 문제와 성능 튜닝:**
   - *상황:* 유저 랭킹보드에서 100명의 유저 목록을 조회할 때, 각 유저의 '최근 거래 내역 1건'을 가져오기 위해 쿼리가 101번 나가는 N+1 문제 발생.
   - *해결:* JPA Fetch Join 및 DTO 직접 조회를 통해 쿼리를 1~2번으로 최적화.
